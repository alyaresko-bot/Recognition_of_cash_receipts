# Телеграм-бот для распознавания чеков и записи в Google Sheets

### Описание

Телеграм-бот:

- принимает **одно или несколько фото** одного кассового чека;
- при наличии на чеке **фискального QR** (ФНС) извлекает из него дату, время, ФН, ФД, фискальный признак и итог; при неудаче использует только распознавание по фото;
- распознаёт содержимое чека **vision-моделью OpenAI** (например, GPT‑4o-mini) по системному промпту из `system_prompt.txt`;
- возвращает **структурированный JSON** и **записывает данные в Google Sheets** (агрегат по чеку и позиции на отдельном листе).

Распознавание реализовано в `ocr.py`: перед отправкой в API каждое изображение проходит **предобработку** (`image_preprocess.py`): автоповорот по **EXIF Orientation** и ограничение длинной стороны. Для длинных чеков бот ожидает последовательные кадры **без пересечений позиций** между соседними фото и требует вернуть **все позиции без пропусков и обрезания строк**. Различия в составе/ингредиентах должны приводить к отдельным строкам в `items`.

### Пайплайн распознавания (актуальная логика)

1. **Предобработка** каждого кадра (`image_preprocess.py`) — те же файлы, что затем уходят в vision API.
2. **Фискальный QR** (`receipt_qr.py`, опционально): по кадрам ищется QR; сначала используется **QReader** (если пакет установлен и инициализируется), иначе или при неудаче — **OpenCV `QRCodeDetector`**. Если строка соответствует формату данных ФНС (поля `t`, `s`, `fn`, `i` или `fd`, `fp`), из неё извлекаются дата, время, номер ФН, номер фискального документа (уникальный номер чека в разрезе ФН), фискальный признак (`fp`) и итоговая сумма (`s` в копейках, в JSON/таблицу — в рублях с запятой).
3. **Vision (OpenAI)** по системному промпту `system_prompt.txt`: позиции товаров и остальные поля читаются с изображений. Если QR был успешно разобран, в пользовательский запрос добавляется подсказка с эталонными полями; после ответа модели значения **дата, время, номер чека (ФД), фискальный признак, итог и номер ФН** из QR **принудительно перезаписывают** соответствующие поля в `receipt_info` (чтобы в таблицу попали данные из QR, а не галлюцинации модели).
4. Если QR **не** прочитан или строка **не** распознана как фискальная, обработка идёт **только по старой схеме** (только vision и промпт).
5. **Повторный запрос к LLM**: если первая попытка не прошла внутреннюю проверку сумм и позиций, выполняется вторая попытка. Флаг «нужна ручная проверка» для бота и таблицы (`_meta.needs_review`) выставляется **по итоговому** JSON после всех попыток и слияния с QR, а не по результату первой попытки — чтобы не показывать предупреждение, когда вторая попытка уже всё исправила.

Метаданные ответа `analyze_receipt_image`: `_meta.qr` — `used` (был ли успешный фискальный QR), при успехе также `decoder` (`qreader` / `opencv`) и `raw` (сырая строка QR).

Отключить этап QR:

```env
RECEIPT_QR_ENABLED=0
```

Пакет **`qreader`** тянет тяжёлые зависимости (в т.ч. модель для поиска QR); при первом запуске может потребоваться интернет для загрузки весов. Если QReader недоступен, используется только OpenCV — без отдельной настройки.

### Структура проекта

| Файл | Назначение |
|------|------------|
| **`bot.py`** | Telegram: команды, меню, режимы загрузки, напоминания, обработка фото и колбэков |
| **`config.py`** | Загрузка настроек из `.env` |
| **`ocr.py`** | Предобработка, опционально фискальный QR, вызов OpenAI Vision, чтение `system_prompt.txt`, поддержка одного или нескольких файлов изображения |
| **`receipt_qr.py`** | Декодирование QR (QReader / OpenCV), разбор строки ФНС, подсказка для LLM и слияние полей в `receipt_info` |
| **`image_preprocess.py`** | EXIF-поворот, уменьшение по длинной стороне, обрезка полей перед vision |
| **`google_sheets.py`** | Запись чека и позиций в Google Sheets |
| **`system_prompt.txt`** | Системный промпт для LLM (формат JSON, правила полноты распознавания по чеку) |
| **`receipt_state_keys.py`** | Ключи состояния сессии загрузки чека |
| **`receipt_state_store.py`** | Персистентность состояния: **SQLite** (по умолчанию) или **Redis** |
| **`requirements.txt`** | Зависимости Python (`python-telegram-bot[job-queue]`, `qreader`, `aiosqlite`, `redis`, `pillow`, `opencv-python-headless`, …) |
| **`.env.example`** | Пример переменных окружения |

### Подготовка окружения

1. **Python 3.10+**

2. **Виртуальное окружение** (PowerShell), подставьте свой путь к каталогу проекта:

   ```powershell
   cd путь\к\Chek-bot
   python -m venv .venv
   .\.venv\Scripts\Activate.ps1
   ```

3. **Зависимости** (включая **JobQueue** для отложенных напоминаний):

   ```powershell
   pip install -r requirements.txt
   ```

### Настройка Telegram и OpenAI

1. Создайте бота в BotFather, получите **токен**.
2. Скопируйте `.env.example` в `.env` и заполните минимум:

   ```env
   TELEGRAM_BOT_TOKEN=ваш_токен_от_BotFather
   OPENAI_API_KEY=ваш_openai_api_key
   OPENAI_VISION_MODEL=gpt-4o-mini
   # Необязательно: перекроет OPENAI_VISION_MODEL
   # OPENAI_MODEL=gpt-4o-mini
   ```

3. **Google Sheets** (таблица и сервисный аккаунт):

   ```env
   GOOGLE_SHEETS_SPREADSHEET_ID=идентификатор_таблицы
   GOOGLE_SERVICE_ACCOUNT_FILE=credentials.json
   GOOGLE_SHEETS_SHEET_NAME=Sheet1
   GOOGLE_SHEETS_ITEMS_SHEET_NAME=Товар
   ```

### Персистентность состояния загрузки чека

Состояние незавершённой загрузки (режим, список `file_id` фото, флаги ожидания и т.д.) хранится **вне оперативной памяти процесса**, чтобы пережить перезапуск бота.

**SQLite (по умолчанию)** — файл рядом с проектом:

```env
RECEIPT_STATE_BACKEND=sqlite
# необязательно, по умолчанию receipt_state.db в корне проекта
# RECEIPT_STATE_SQLITE_PATH=receipt_state.db
```

**Redis** — если несколько инстансов бота или отдельный сервер:

```env
RECEIPT_STATE_BACKEND=redis
REDIS_URL=redis://localhost:6379/0
```

### Настройка Google Sheets

1. Два листа:
   - **Лист чеков** (`GOOGLE_SHEETS_SHEET_NAME`), колонки **A–P**:
     - **A–N** — как раньше: organization, inn, date, time, receipt_number, shift_number, fiscal_sign, payment_type, total_discount, total_amount, vat, items_count, validation_status, validation_comment.
     - **O** — номер фискального накопителя (**ФН**), подставляется при успешном разборе фискального QR; если QR не использовался, ячейка пустая.
     - **P** — **«Да»** / **«Нет»**: распознан ли фискальный QR и применён ли патч (`_meta.qr.used`). Удобно для фильтрации без анализа столбца O.
   - **Лист товаров** (`GOOGLE_SHEETS_ITEMS_SHEET_NAME`): receipt_id, item_index, name, quantity, unit_price, item_discount, item_total. Первая строка — заголовки.

   Рекомендуется добавить в первую строку листа чеков подписи для **O** (например, «ФН (QR)») и **P** (например, «QR распознан»).

2. В Google Cloud: проект, **Google Sheets API**, **Service Account**, JSON ключа в файл `credentials.json` (или путь в `.env`).

3. Выдайте таблице доступ на запись e-mail сервисного аккаунта.

### Системный промпт

Файл **`system_prompt.txt`** задаёт формат JSON, обязательные поля, требования к качеству фото и правила для **нескольких снимков одного чека**: извлечение всех позиций и консервативное разделение вариантов товара по различиям состава/ингредиентов в `items`. Редактирование промпта не требует правок в `bot.py`; пользовательское сообщение в `ocr.py` для мультифото дополнительно напоминает модели о самопроверке по суммам и о необходимости возвращать полный список позиций без пропусков.

### Предобработка изображений

В `.env` можно отключить шаги или изменить лимит размера:

```env
# по умолчанию включено (1 / true). Для отключения: 0 / false / no / off
RECEIPT_PREPROCESS_ENABLED=1
# максимум длинной стороны в пикселях после поворота (до обрезки; по умолчанию 3000)
RECEIPT_PREPROCESS_MAX_SIDE=3000

# Обрезка полей по контурам (эвристика).
# Если видите, что LLM переносит числа между строками — отключите (0).
# RECEIPT_PREPROCESS_CROP_ENABLED=0
```

Если `RECEIPT_PREPROCESS_CROP_ENABLED=1`, обрезка строится по адаптивному порогу и контурам (основной метод) и при необходимости по «небелому» фону (запасной). Если уверенная область не найдена или она слишком мала относительно кадра, сохраняется кадр после **поворота и масштабирования** без агрессивной обрезки.

### Как пользоваться ботом в Telegram

После **`/start`** доступна **нижняя клавиатура**:

- **Один чек (одно фото)** — один снимок или несколько: после **каждого** фото бот спрашивает, **крайнее ли это фото чека** (кнопки под сообщением).
- **Длинный чек** — несколько фото подряд **сверху вниз**; завершение: кнопка **«Загрузка завершена»** или текст с фразой **«Загрузка завершена»**.
- **Помощь** — краткая справка.

Можно отправить фото **без выбора режима** — используется режим «один чек».

**Рекомендация по съёмке** (дублируется в текстах бота): если чек не помещается в кадр, снимайте частями строго сверху вниз; каждый следующий снимок должен начинаться со **следующей новой позиции**, без повторения строк с предыдущего кадра.

**Напоминание:** если загрузка не завершена (не ответили «крайнее фото», не нажали завершение длинного чека и т.д.), примерно через **5 минут** после последнего действия бот пришлёт напоминание.

**Незавершённая загрузка:** если при ожидании ответа «крайнее фото?» приходит **ещё одно фото**, бот предупреждает о незавершённом чеке (с временем начала) и предлагает кнопки **«Продолжить чек»** / **«Начать новый»**. Аналогично при смене режима через меню при уже активной загрузке — **«Продолжить текущий»** / **«Начать новый»**.

Фото нужно отправлять **как изображение**, не как документ-файл (если не настроено иное).

### Запуск

Запускайте бота **интерпретатором из виртуального окружения**, где установлены зависимости (иначе возможна ошибка `No module named 'telegram'` при вызове системного `py -3.11` без venv):

```powershell
cd путь\к\Chek-bot
.\.venv\Scripts\Activate.ps1
python bot.py
```

Либо без активации:

```powershell
.\.venv\Scripts\python.exe .\bot.py
```

Бот работает в режиме **long polling**.

### Перед публикацией на GitHub

1. **Секреты не в репозитории.** В корне проекта добавлен `.gitignore`: игнорируются `.env`, `.venv/`, `credentials.json`, типичные имена JSON-ключей сервисного аккаунта (`receipt-*.json`, `*-service-*.json`), локальная БД `receipt_state.db`. Перед первым `git push` выполните `git status` и убедитесь, что в коммит не попали файлы с `private_key` или токены.
2. **Если ключ Google уже лежал в каталоге с именем вроде `receipt-*-*.json`**, не добавляйте его в Git. При любом сомнении, что файл мог утечь, в Google Cloud Console **отзовите старый ключ** сервисного аккаунта и создайте новый; храните JSON вне репозитория (или локально как `credentials.json`, путь — в `.env`).
3. **Проверка:** `git check-ignore -v путь\к\файлу` — убедиться, что секретный файл попадает под правило ignore.
4. **Локальные заметки** (например, `НОВАЯ РЕДАКЦИЯ.txt`) и каталог **`.idea/`** в публичный репозиторий обычно не кладут — при желании добавьте имя файла в `.gitignore` или удалите из корня перед публикацией.
5. Первый коммит после добавления `.gitignore`: если секреты уже попадали в историю Git, одного ignore недостаточно — нужно [очистить историю](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository) или создать новый репозиторий без старых коммитов.

### Доработка под себя

- **Промпт и поля JSON** — правьте `system_prompt.txt` и при необходимости колонки/логику в `google_sheets.py`.
- **Фискальный QR** — разбор строки и поля ФНС задаются в `receipt_qr.py`; отключение этапа — `RECEIPT_QR_ENABLED` в `.env`.
- **Таблица** — расширьте `append_receipt_row` и схему листов под новые поля.
- **Модель** — переменные `OPENAI_VISION_MODEL` / `OPENAI_MODEL` в `.env`.
